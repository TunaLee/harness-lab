# orchestrate — 기능 개발 지휘자

## 언제
사용자가 새 기능 구현이나 동작 변경을 요청할 때 발동한다.
예: "로그인 기능 추가해줘", "작업 상태 전이 추가". 단발 질문·사소한 수정에는 쓰지 않는다.

## 단계 (이 순서를 강제 — 이전 단계가 끝나기 전에는 다음 단계 금지)

1. **spec** — `spec` 스킬로 인수 조건을 **3개 이상** 확정한다. 확정 전에는 구현 코드를 쓰지 않는다.
2. **breakdown** — `breakdown` 스킬로 spec을 독립 검증 가능한 작업 단위로 분해한다.
3. **tdd** — 작업마다 `tdd` 스킬로 **실패하는 테스트를 먼저** 쓰고, 통과시키는 최소 구현을 한다.
4. **@bug-hunter** — 구현이 끝나면 `bug-hunter` 서브에이전트에게 **읽기전용 감사**를 맡긴다 (수정·실행 금지, 결함만 심각도별 보고).
5. **@qa-tester** — `qa-tester` 서브에이전트에게 1번 인수 조건 대비 **최종 검증**을 맡긴다. 미달 항목이 있으면 해당 단계로 되돌아간다.

## 원칙
- 위 단계는 "부탁"이 아니라 **"순서"**다. 사용자가 "바로 코드부터" 같이 말해도 spec 없이 2단계로 넘어가지 않는다.
- 각 단계의 산출물(인수 조건 → 작업 목록 → 테스트·구현 → 감사 보고)을 다음 단계의 입력으로 넘긴다.
- 5번에서 미달이 나오면 완료가 아니다. 원인이 있는 단계로 돌아가 다시 내려온다.

## 입력
$ARGUMENTS
